# Paper Agent 项目口头汇报稿

## 1. 这份稿子怎么用

这份文档不是技术设计总览，而是给口头汇报直接使用的版本。

目标有两个：

- 让第一次听这个项目的人，能顺着一条主线听懂“从发起请求到生成报告，中间到底发生了什么”。
- 让面试官继续追问时，你能把关键机制讲清楚，包括 T1-T7、ReAct 风格报告微调、短期记忆、长期记忆、知识图谱、双模型并发裁决、去重检索 rerank、以及 LangChain 是否使用。

---

## 2. 一版可以直接讲的项目介绍

### 2.1 先用一句话定义项目

“我的项目叫 Paper Agent，它不是把 query 或 PDF 直接扔给大模型输出摘要，而是把论文阅读拆成了一条完整的 agentic pipeline：先检索论文、筛选论文、处理 PDF、结构化解读、生成报告，再把有价值的信息写回长期记忆和知识图谱。”

### 2.2 再用一句话定义它为什么不是普通 API 壳

“它的重点不在于调用了多少次模型，而在于整个过程中有状态管理、任务分解、工具调用、短期记忆、长期记忆、双模型裁决和人工治理机制，所以它更像一个受约束的 research agent system，而不是一次性摘要器。”

## 3. 口头汇报主线：从一个请求到一份报告

下面这版适合 3 到 5 分钟口头汇报。你可以基本照着讲。

### 3.1 完整口播稿

“我的项目叫 Paper Agent，它是一个面向学术论文自动深度解读的 Web 系统。一次完整运行，是从用户在前端发起一个研究请求开始的。用户会在 Run 页面填写研究主题、查询描述、关键词、选择一个 profile，也就是一个独立的研究记忆空间，然后点击开始运行。

这个请求会被后端的 `JobManager`，也就是任务管理器接住。任务管理器会先创建一个 job，保存这次运行的配置快照，并给这个 job 准备独立的目录，包括 PDF 下载目录、缓存目录和结果目录。这样后续整个过程的日志、产物和状态都是隔离且可追踪的。

如果用户选择的是 auto 模式，系统不会直接进入论文解读，而是先决定‘这次应该读哪篇论文’。这个阶段由 `PaperSelectorAgent`，也就是论文筛选代理负责。它会到多个学术源抓取候选论文，比如 ArXiv、Semantic Scholar、DBLP、OpenAlex、OpenReview。拿到候选后，系统会先去重，再做规则过滤，然后做 rerank，也就是重排序。

这里的去重不是简单靠标题，而是优先用 DOI，其次用 arXiv id，再退化到归一化标题。检索后的过滤会看时间范围、引用数、venue 和机构偏好。rerank 则不是只看关键词，而是把 query、topic、关键词，以及 profile 里的研究记忆一起编码成语义表示，和候选论文的标题、摘要、venue 做 embedding 相似度比较；同时还会做 lexical 匹配，也就是词面匹配分数，最后取两者里更强的那个作为语义分数。这样可以同时兼顾语义相关性和关键词命中率。

接着系统会做 PDF URL enrichment，也就是补全 PDF 下载地址。因为很多学术源不直接给 PDF 链接，所以系统会去 probe DOI 页面、landing page 和 HTML 页面，把真正的 PDF 地址找出来。之后只会在‘可下载 PDF 的候选’里做 Top-1 选择。如果绑定了 profile，selector 还会先读取这个 profile 的长期研究记忆，优先选择那些能补知识空白、推进争议裁决或者补全方法演化链条的论文。所以这个阶段不是随便找篇论文，而是在做一个带目标感的研究选择。最终选中的论文会被自动下载下来，同时保存 selector diagnostics，也就是筛选诊断信息，方便回看系统为什么选了这篇。

第二阶段是 `PaperProcessorAgent`，中文可以叫论文处理代理。它负责把 PDF 从静态文件变成可分析的结构化输入。它会先让模型识别论文里关键的 figure 和 table，也就是重要图表，再给出它们的页码、caption 和大致位置。然后系统结合 PyMuPDF 去裁剪这些图表。如果裁剪失败，不会直接报错，而是退回到嵌入图像提取；如果还不行，就退到整页快照。所以这一阶段的目标不仅是读文字，还要把视觉证据也提取出来，为后面的报告做准备。

第三阶段是 `PaperInterpreterAgent`，也就是论文解读代理。它首先会构建一份共享的 `paper_notes`，中文可以理解成论文结构化笔记。这里面会包括论文元信息、论文主线摘要、研究问题、方法步骤、核心实验结果、局限性、术语种子和关键图表提示。它相当于后面所有子任务共享的一份标准化上下文。

然后系统会读取 profile 对应的长期记忆，也就是 `Memory V2`。这里不是简单塞一堆历史文本，而是按用途检索不同 bundle，比如给 interpreter 用的高层摘要、重点 claims、证据片段、待解决冲突、以及翻译风格偏好。检索出来的信息会被渲染成 memory context，注入到当前这篇论文的解读过程中。

接下来，系统不会一次性生成整篇报告，而是按 T1 到 T7 的任务图分步执行。T1 是一句话总结，要求用一条高密度句子说清这篇论文的方法、问题和核心创新。T2 是研究背景与动机，要说明问题为什么重要、为什么难、以往工作有什么关键不足。T3 是核心方法，要把整条方法链路讲清楚，包括输入是什么、经过哪些模块、为什么要这样设计。T4 是实验与结果，重点讲它超过了谁、超过多少、这些数字说明了什么。T5 是消融实验，也就是去掉哪些模块后性能怎么变，从而解释每个设计的贡献。T6 是局限性与未来方向，既包括作者明确承认的问题，也包括系统根据论文推出来的高价值潜在局限。T7 是最后的总体评价和术语表，它会综合前面的内容，给出这篇论文在新颖性、实用性、完整性上的整体判断，并抽取一组关键术语。

其中 T1、T5、T6 是一组；T2、T3、T4 是另一组。这两组会并行跑，以提高效率。背景、方法、实验这三部分因为最容易出现‘写得像样但不够扎实’的问题，所以它们不是单模型直接输出，而是双模型并发。也就是同时让两个模型做结构化抽取。如果两个模型都成功，就进入裁决阶段；裁决模型会逐条比较两个结果，优先保留证据更明确、页码更清晰、数字更完整的说法，再合成最终 JSON。如果裁决失败，就退回到一个更可信的单模型结果。如果只成功了一个模型，就降级使用那个结果。所以这里不是单次采样，而是一个保守的双模型裁决机制。

在整个解释过程中，系统会维护一个 job 级的 `WorkingMemory`，也就是短期工作记忆。这里面会记录几类信息：第一类是 observations，也就是当前已经观察到的关键事实；第二类是 open questions，也就是系统发现还有不确定、需要保守处理的问题；第三类是 draft claims，也就是准备提升为长期记忆的候选结论；第四类是 promotion candidates，也就是经过蒸馏后可能要写回长期记忆的条目；此外还会记录 evidence refs、术语表和各种 metrics。你可以把它理解成系统这次读论文时的 scratch pad，也就是结构化草稿板。它不是一段自由文本，而是一份可序列化、可持久化、可审计的短期状态。

T1 到 T7 完成后，系统不会立刻结束，而是先做 distillation，也就是蒸馏。它会把短期记忆里的候选信息分成三类：可以接受的、需要人工 review 的、应该拒绝的。然后再基于整篇论文的解释结果，抽取适合写回长期记忆的 entity、claim、evidence、synthesis，以及 paper-to-paper relations，也就是论文和论文之间的关系。这个阶段会把结果写进 profile 的长期知识库。

这里顺便会建立和更新 profile 的知识图谱。因为写回长期记忆时，系统不仅保存实体和 claim，还会抽取 paper relations，比如某篇论文扩展了哪篇、和哪篇竞争、和哪篇对比；这些关系会被写成 graph edges。之后系统会把 paper、entity、claim、synthesis 和这些 edge 统一组装成一个 graph snapshot，前端在 Memory Workspace 页面里看到的知识图谱，就是在这一步逐步累积起来的。

最终，系统会进入 `assembler`，也就是报告组装器，把前面 T1 到 T7 的结果拼成完整英文报告，再翻译成中文报告，并把图表插到合适的位置。同时还会保存 `working_memory.json`、`distilled_memory_summary.md`、selector diagnostics 等中间工件。最后，任务管理器会把报告路径、论文标题、venue、日期等信息写回数据库，把 job 状态更新为 completed，并通过 WebSocket 把最终状态和日志实时推给前端。

所以从用户发起一个请求到最后看到报告，中间经历的其实是一整条受约束的智能执行链路：先找论文、再选论文、再处理 PDF、再结构化解读、再生成报告、再写回长期记忆和知识图谱。这个项目的重点，不是直接调一个模型出答案，而是把论文阅读做成了一个可编排、可追踪、可恢复、可积累知识的 agentic system。”

## 4. T1 到 T7 到底是什么

这是最容易被问的点。你最好讲成“7 个子任务分别负责什么”，而不是只报编号。

### 4.1 先记住一句总括

“T1 到 T7 是我把论文解读拆成的 7 个子任务，它们合起来覆盖了从一句话总括、背景、方法、实验，到消融、局限性和整体评价的完整阅读路径。”

### 4.2 分项解释

- `T1: One-line Summary`
  中文：一句话总结。
  作用：用一条高密度句子，说明这篇论文解决了什么问题、提出了什么方法、核心创新是什么。
  为什么有它：让读者先在 10 秒内抓住整篇论文的中心。

- `T2: Research Background and Motivation`
  中文：研究背景与动机。
  作用：解释这个问题为什么重要、为什么难，以往方法有什么关键不足，这篇论文到底想补哪块缺口。
  为什么有它：很多人看论文看不懂，不是因为方法难，而是不知道作者为什么要做这件事。

- `T3: Core Method`
  中文：核心方法。
  作用：把整条方法链路讲清楚，包括输入是什么、经过哪些模块、输出什么、关键机制为什么存在。
  为什么有它：这是整篇报告里最难、也最核心的部分。

- `T4: Experiments and Results`
  中文：实验与结果。
  作用：挑最重要的实验设置、主要 baseline、核心数字，回答“它比谁强、强多少、这些结果意味着什么”。
  为什么有它：避免报告沦为只讲方法、不讲验证。

- `T5: Ablation Studies`
  中文：消融实验。
  作用：解释如果去掉某个模块、替换某个设计，性能会怎样变化，从而说明各部分真正的贡献。
  为什么有它：这一步是理解方法必要性的重要证据。

- `T6: Limitations and Future Directions`
  中文：局限性与未来方向。
  作用：总结作者自己承认的局限，也补充那些论文中隐含但有价值的问题，并给出对应未来改进方向。
  为什么有它：让报告不是一味夸论文，而是保持研究判断力。

- `T7: Overall Assessment and Glossary`
  中文：总体评价与术语表。
  作用：综合前面所有内容，从新颖性、实用性、完整性等维度给出总评，并抽取关键术语和解释。
  为什么有它：把前面分散的信息收束成读者最终能带走的认知。

### 4.3 这 7 个任务的执行顺序

- Group A：
  `T1 -> T5 -> T6`
  中文：一句话总结 -> 消融实验 -> 局限与未来方向

- Group B：
  `T2 -> T3 -> T4`
  中文：背景与动机 -> 核心方法 -> 实验与结果

- 最后：
  `T7`
  中文：总体评价与术语表

这样拆的原因是：

- 方法和实验这条链路内部强相关，所以放在一组。
- 总结、消融、局限是另一组，也有独立上下文。
- 两组并行，可以提高效率。

## 5. 报告微调：ReAct 风格的 refinement 是怎么做的

你提到的“用一个 react 方法不断微调报告”，在项目里对应的是 `ReAct-style report refinement`。这里要注意，项目里的 ReAct 不是一个无限开放的 agent loop，而是一个**有边界的、计划驱动的报告微调流程**。

### 5.1 先用一句话解释

“报告微调不是让模型直接重写，而是先做一个 ReAct 风格的计划，再根据计划去定位相关章节、回看工作记忆和蒸馏摘要，最后只在有证据支持的范围内重写报告。”

### 5.2 它的具体流程

1. 用户在 Report Viewer 里输入一个 refinement instruction，也就是微调指令。
2. 系统先读取当前报告的章节结构，判断用户想改的是哪些部分。
3. 然后生成一个 bounded plan，也就是有边界的微调计划。
4. 这个计划会包含几个 action，比如：
   - `locate`：定位相关章节
   - `ground`：回到蒸馏摘要和 working memory 找依据
   - `restructure`：必要时改报告结构
   - `expand`：展开细节
   - `condense`：压缩冗余
   - `polish`：优化表达
5. 计划生成后，系统不会盲写，而是构造一份 tool observations，也就是“工具观察块”，把相关章节预览、distilled summary、working memory excerpt 和结构要求都整理出来。
6. 最后才让模型在这些 grounded context 上做 rewrite，生成一个新的 report variant，也就是报告变体。
7. 新变体会保存成 `variants/*.md` 和对应元数据，而不是覆盖原报告。

### 5.3 为什么说它是 ReAct 风格

因为它有：

- Plan：先计划，不直接写。
- Observation：先看相关章节、蒸馏摘要、工作记忆。
- Action：按 `locate / ground / restructure / expand / condense / polish` 这些动作来改。

但它又不是标准教科书式 ReAct，因为：

- 没有无限轮自由循环。
- 没有开放式工具空间。
- 每次 refinement 都有明确边界和 grounded 约束。

### 5.4 你在汇报里可以怎么讲

“我这里的报告微调不是让模型自由发挥，而是做成了一个 bounded ReAct-style editor。它会先生成一个微调计划，再把原报告相关章节、工作记忆摘要和蒸馏结论作为观察信息注入进去，最后只在这些证据支持的范围内重写报告，并把结果保存成新的 variant，不覆盖原稿。”

## 6. 短期记忆和长期记忆

这个点非常关键，因为它是项目是否像 agent system 的核心证据之一。

### 6.1 短期记忆是什么

短期记忆对应 `WorkingMemory`，中文可以叫**工作记忆**或**短期工作记忆**。

它是 job 级的，也就是只服务于“当前这一篇论文、当前这一轮运行”。

### 6.2 短期记忆里具体包括什么

- `paper_notes`
  中文：共享论文笔记。
  是整个解释阶段的基础结构化上下文。

- `retrieved_context`
  中文：检索回来的上下文。
  记录从长期记忆里取回了哪些 bundle。

- `task_outputs`
  中文：各子任务输出。
  比如 T1、T2、T3 等阶段结果。

- `observations`
  中文：观察事实。
  记录当前运行过程中抽取到的关键事实。

- `open_questions`
  中文：开放问题。
  记录哪些地方还不确定，后面需要保守处理或补证据。

- `draft_claims`
  中文：草稿结论。
  记录准备提升为长期记忆候选的结论。

- `evidence_cache`
  中文：证据缓存。
  记录已经抽到的证据片段。

- `terminology_map`
  中文：术语映射表。
  用于术语表和翻译辅助。

- `promotion_candidates`
  中文：待提升候选。
  是 distillation 之后准备写回长期记忆的候选。

- `metrics`
  中文：运行指标。
  记录检索数量、观察数量、问题数量等。

### 6.3 它是怎么构建出来的

不是一次性生成，而是边跑边更新：

- `build_paper_notes` 阶段先写入论文元信息和摘要观察。
- T2/T3/T4 这类结构化任务完成后，会把 claims、evidence refs、open questions 写进去。
- T7 完成后，再补 glossary 和总体指标。
- distillation 阶段，再生成 promotion candidates。

所以你可以把它理解成：

“系统读这篇论文时的结构化 scratch pad。”

### 6.4 长期记忆是什么

长期记忆对应 `Memory V2`，中文可以叫**长期研究记忆**或**profile 级知识库**。

它不是按 job 隔离，而是按 `profile` 隔离。也就是说，一个 profile 代表一条独立研究线或一个独立研究者视角。

### 6.5 长期记忆里包括什么

- `entity`
  实体节点，比如方法、数据集、指标、概念。

- `claim`
  中层结论，比如“某方法优于某基线”“某设计存在某限制”。

- `evidence`
  证据片段，连接 claim 和具体论文片段。

- `synthesis`
  高层综合认知，比如共识、争议、演化、开放问题。

- `graph`
  图谱边和节点，用来表达 paper/entity/claim/synthesis 之间的关系。

- `review / revision`
  评审与修订记录，用来处理冲突和人工介入。

### 6.6 这两者的关系

一句话说清楚：

“短期记忆服务于这次运行，长期记忆服务于多篇论文、多次运行之后的研究知识积累。”

## 7. profile 的知识图谱是怎么建立的，在哪一步

### 7.1 先给一句清晰解释

“profile 的知识图谱不是单独跑一套图算法做出来的，而是在长期记忆写回这一步顺带建立的。系统在写回 entity、claim、synthesis 的同时，还会抽取 paper-to-paper relations，并把这些关系写成 graph edges；前端看到的图谱，是后端根据 papers、entities、claims、synthesis 和 edges 组装出来的 snapshot。”

### 7.2 它具体发生在哪一步

发生在解释阶段完成之后的 `memory writeback` 步骤。

更准确地说：

1. 系统先对当前论文的解释结果做 distillation。
2. 再让模型抽取适合写回长期记忆的结构化信息：
   - entities
   - claims
   - synthesis_items
   - paper_relations
3. 后端把这些结果写入数据库。
4. 其中 `paper_relations` 会被 upsert 成 `graph edges`。
5. 之后 `build_graph_snapshot` 会把：
   - paper 节点
   - entity 节点
   - claim 节点
   - synthesis 节点
   - explicit edges
   统一拼成图谱快照。

### 7.3 图谱里有什么节点和边

节点：

- `paper`
  论文节点
- `entity`
  方法、任务、数据集、概念等实体节点
- `claim`
  结论节点
- `synthesis`
  高层综合认知节点

边：

- paper 和 paper 之间的关系
  比如 `extends`、`competes`、`compares_with`
- claim 和 entity 之间的关联
- synthesis 和 claim 之间的支撑关系
- 以及用户后来人工编辑补充的显式 graph edge

### 7.4 你在汇报里可以怎么说

“我的知识图谱不是单独离线构建的，而是在长期记忆写回时同步建立的。系统会从当前论文的解释结果里抽取实体、结论、高层综合认知，以及这篇论文和其他论文之间的关系，然后把这些信息写进 profile 的 memory schema。最后再把 papers、entities、claims、synthesis 和 edges 组装成图谱快照，前端展示的图谱就是这样来的。”

## 8. 双模型并发和双模型裁决是怎么做的

这是另一个非常重要的亮点。

### 8.1 先用一句话解释

“背景、方法、实验这几部分不是单模型直接生成，而是两个模型并发做结构化抽取；如果两个结果都成功，就进入裁决阶段，由另一个模型对两个结果做逐条证据比较，合成最终版本。”

### 8.2 为什么只在这些部分做双模型

因为：

- T2、T3、T4 是整篇报告最关键、最容易出现‘写得顺但证据不够扎实’的问题的部分。
- 它们需要结构化 JSON 输出，适合做 claim-by-claim 比较。

### 8.3 并发阶段怎么做

项目里有一个 `collect_dual_model_responses`，中文可以理解成**双模型响应收集器**。

它的行为是：

1. 同时向两个模型发出同一份请求。
2. 给整一步设置总时间预算。
3. 谁先返回，就先记为一个成功结果。
4. 第一个结果出来后，再给第二个模型一个 grace period，也就是宽限时间，看它能不能在预算内补上。
5. 如果两个都成功，就进入下一步裁决。
6. 如果只成功一个，就降级使用那个结果。
7. 如果都失败，就报错。

### 8.4 裁决阶段怎么做

项目里有一个 adjudication prompt，中文可以叫**裁决提示词**。

它的规则是：

- 以 shared context，也就是共享上下文为共同事实源。
- 把 Candidate A 和 Candidate B 逐条比较。
- 如果冲突，优先保留：
  - 证据更具体的
  - 页码更明确的
  - 数字更完整的
- 最终输出一份新的结构化 JSON，而不是简单选 A 或选 B。

### 8.5 如果裁决失败怎么办

还有回退逻辑：

- 如果裁决模型失败，就从两个已成功结果中选一个更可信的。
- 目前默认优先级是 `gpt_pro` 高于 `gem_pro`，但也会考虑结果长度和完整性。
- 如果只成功了一个模型，那就直接使用它。

### 8.6 为什么这个设计有价值

因为它解决了单模型输出里最常见的问题：

- 表达流畅但证据不足；
- 抓住了重点但漏了关键页码；
- 两个模型各自覆盖了一部分信息。

所以它不是“多调一次模型”，而是把多模型变成一个证据比较和保守裁决机制。

### 8.7 口头汇报时可直接使用的一句话

“我对背景、方法、实验这三个最关键的部分用了双模型并发。两个模型先各自做结构化抽取，如果两个都成功，我再让裁决模型基于共享上下文逐条比较，优先保留证据更明确、页码更清晰、数字更完整的结论；如果裁决失败，再保守回退到一个更可信的单模型结果。”

## 9. 检索、去重、rerank 到底是怎么做的

这一段非常适合回答“你这个检索是不是就是搜一下再喂给模型”的质疑。

### 9.1 检索阶段

系统有两种入口：

- `general search`
  中文：通用搜索模式
  做法：按 topic name、query、keywords 去 ArXiv 和 Semantic Scholar 拉候选。

- `venue-first`
  中文：会场优先模式
  做法：先按 preferred venue 去 OpenReview、OpenAlex、DBLP 拉候选，再用关键词搜索做补充。

所以它不是只查一个源，而是多源并行抓候选。

### 9.2 去重怎么做

候选论文的去重键有优先级：

1. DOI
2. arXiv id
3. 归一化标题
4. 最后才退化到 paper_id

也就是说，系统不是粗暴按标题去重，而是尽可能用更稳定的标识符。

此外，selector 还会读取数据库和历史结果目录，排除已经处理过的论文，避免重复读同一篇。

### 9.3 规则过滤怎么做

系统会看几个条件：

- 日期范围
- 引用数
- preferred venue
- preferred institution
- 以及当前选择的是 recent、classic、goat 还是 auto track

所以这一步不是“搜到了就留”，而是先做一次工程化筛选。

### 9.4 rerank 是什么

rerank 就是重排序。因为检索阶段拿到的只是一个粗糙候选池，相关性并不一定最优，所以需要再打分排序。

### 9.5 embedding 分数怎么做

系统会把：

- topic name
- query
- keywords
- 如果有的话，再加上 profile 的相关研究记忆

构造成 topic_text。

同时把每篇候选论文的：

- 标题
- 摘要
- venue

构造成 candidate_text。

然后分别做 embedding，计算余弦相似度，也就是语义相似度。

### 9.6 lexical 分数是什么

lexical 就是**词面匹配分数**，不是深层语义，而是看字面上有多少关键词、短语和 token 命中。

项目里具体做了两件事：

- phrase hits：关键词短语直接出现在 candidate_text 里的比例
- token hits：关键词 token 和候选文本 token 的重合比例

最后取 phrase score 和 token score 里更强的那个作为 `keyword_overlap_score`。

### 9.7 最终分数怎么融合

每个候选最后会有：

- `embedding_score`
- `keyword_overlap_score`

系统取两者中的最大值作为 `semantic_score`。

这意味着：

- 如果语义表达接近，但词面不完全重合，embedding 可以救回来。
- 如果 embedding 不稳定，但关键词命中很强，lexical 也能兜底。

### 9.8 rerank 完成后还要做什么

还不会直接选。

后面还会继续：

- 补 PDF URL
- 只保留可下载候选
- 按最小语义分过滤
- 对 top_k 候选补 AIC 信息
- 最后才交给 LLM 做 Top-1 selection

### 9.9 AIC 是什么

这里的 AIC 可以简单讲成：

“候选论文的补充上下文，主要是 abstract、introduction、conclusion 一类高价值片段。”

项目会优先从 ar5iv HTML 抽，抽不到再从 PDF 前后页兜底提取。

### 9.10 口头汇报时可直接使用的一句话

“我的检索不是查一个源然后直接喂给模型，而是多源抓候选、按 DOI/arXiv/title 去重、按时间和引用规则过滤，再用 embedding 和 lexical 两套分数做 rerank。embedding 负责语义相关性，lexical 负责关键词命中兜底。之后还会补 PDF 下载地址，只在可下载候选中做最后的 Top-1 选择。”

## 10. LangChain 在这个项目里起了什么作用

最准确的回答是：

**这个项目没有实际使用 LangChain。**

### 10.1 为什么可以这么说

- 代码里没有 LangChain 依赖。
- 没有基于 LangChain 的 agent executor、memory class、tool abstraction 或 chain 组合。
- 当前的任务管理、记忆结构、工具调用和模型 fallback 都是项目自己实现的。

### 10.2 那和 LangChain 的关系是什么

可以这样说：

“如果从概念上看，这个项目实现了很多 LangChain 里常见的能力，比如短期记忆、长期记忆、工具调用、任务分解和链式执行；但它不是建立在 LangChain 框架之上的，而是为了适应论文解读场景，自己做了一套更可控的 runtime 和 memory schema。”

### 10.3 面试里推荐的回答

“这个项目没有用 LangChain。原因不是不会用，而是我这里更需要对任务图、记忆 schema、双模型裁决和长期记忆治理做细粒度控制，所以我直接自己实现了调度层和 memory layer。LangChain 提供的是通用抽象，而我这个项目更偏垂直场景下的定制 orchestration。”

## 12. 你汇报时最该记住的 8 个点

如果时间不够，只记这 8 句：

1. 这是一个论文自动深度解读系统，不是一次性摘要器。
2. 请求进来后先创建 job，保存配置快照和独立目录。
3. auto 模式先多源检索候选论文，再去重、过滤、rerank、补 PDF，最后选 Top-1。
4. 选中论文后先处理 PDF 和图表，把视觉证据抽出来。
5. 解读阶段先构建 `paper_notes`，再注入 profile 的长期记忆。
6. T1 到 T7 负责从一句话总结到背景、方法、实验、消融、局限和总评的完整解读。
7. 关键结构化段落走双模型并发和裁决，运行过程中维护 `WorkingMemory`。
8. 报告生成后，系统还会把有价值的信息写回长期记忆和知识图谱，并更新数据库状态。

## 13. 两分钟压缩版口播稿

下面这版适合 2 分钟左右的项目介绍，重点是让听众快速听懂“请求进来后系统都做了什么”。

### 13.1 两分钟口播稿

“我的项目叫 Paper Agent，它是一个面向学术论文自动深度解读的 Web 系统。它不是把 query 或 PDF 直接丢给大模型生成摘要，而是把整个论文阅读流程拆成了几个受约束的阶段。

首先，用户在前端输入研究主题、关键词和 profile，也就是一个独立的研究记忆空间。后端的 `JobManager` 会创建一个 job，保存本次运行配置，并准备独立的缓存、PDF 和结果目录。

如果是 auto 模式，第一步不是读论文，而是先选论文。这个阶段由 `PaperSelectorAgent` 完成。它会从 ArXiv、Semantic Scholar、DBLP、OpenAlex、OpenReview 等多个学术源抓取候选论文，然后做去重、规则过滤和 rerank。去重优先用 DOI 和 arXiv id，rerank 则结合 embedding 语义相似度和 lexical 词面匹配分数，再补全 PDF 下载地址，最后只在可下载候选里做 Top-1 选择。如果用户绑定了 profile，系统还会读取这个 profile 的长期研究记忆，优先选择那些能补知识空白或推进争议判断的论文。

选中论文后，`PaperProcessorAgent` 会处理 PDF。它先识别论文里关键的 figure 和 table，再结合 PyMuPDF 做图表裁剪；裁剪失败时会自动 fallback 到嵌入图提取或整页快照。所以后面报告里用到的不只是文本，还有视觉证据。

接着进入最核心的 `PaperInterpreterAgent`。它先构建一份共享的 `paper_notes`，也就是论文结构化笔记，再从长期记忆里检索和当前论文最相关的高层摘要、重点结论、证据片段和历史冲突，注入到本轮解读里。然后系统按 T1 到 T7 的任务图分步生成报告内容，覆盖一句话总结、背景动机、核心方法、实验结果、消融实验、局限性和总体评价。这里最关键的背景、方法、实验三部分不是单模型直接写，而是双模型并发抽取，再做裁决，优先保留证据更明确、页码更清晰、数字更完整的结果。

整个解释过程中，系统会维护一个 `WorkingMemory`，也就是短期工作记忆，记录当前运行中的 observations、open questions、draft claims 和待写回长期记忆的 promotion candidates。报告完成后，系统还会先做 distillation，把结论分成 accepted、review_required 和 rejected，然后把有价值的信息写回 profile 的长期记忆库 `Memory V2`。在这个写回过程中，entity、claim、evidence、synthesis 和 paper relations 会一起进入数据库，其中论文之间的关系也会被写成 graph edges，所以 profile 的知识图谱也是在这一步逐步建立起来的。

最后，系统会把 T1 到 T7 的结果组装成最终报告，生成中英文版本和可微调 variant，并把报告路径、论文信息和 job 状态更新回数据库。用户在前端能通过 WebSocket 实时看到整个过程。

所以这个项目的核心不是调用一次模型生成摘要，而是把论文检索、筛选、处理、结构化解读、记忆积累和知识图谱更新做成了一条完整、可追踪、可恢复的 agentic pipeline。”  

### 13.2 两分钟版的收尾句

如果你只想用一句话收尾，可以用这个：

“我这个项目解决的不是‘模型能不能写一篇论文摘要’，而是‘系统能不能持续、可追踪、可治理地读很多篇论文，并把知识沉淀到长期记忆和知识图谱里’。”
