## 1. Profile Detail 500 修复

### 问题现象

访问部分 Profile 详情页时，前端显示 `500 Internal Server Error`。

典型报错接口：

- `GET /api/profiles/{profile_id}/detail`

终端核心异常：

```text
fastapi.exceptions.ResponseValidationError
```

报错字段集中在：

- `response.activity[*].paper_id`
- `response.activity[*].paper_title`
- `response.activity[*].paper_venue`
- `response.activity[*].paper_pub_date`

### 根因分析

`server/database.py` 中的 `list_profile_activity()` 使用了：

```sql
FROM jobs
LEFT JOIN papers ON papers.job_id = jobs.id
```

这意味着只要某个 job 属于该 profile，就会出现在 activity 列表里，即使它没有对应的 `papers` 记录。

在以下场景中，这些 `papers` 字段可能为 `NULL`：

- job 失败，未完成 paper 落库
- job 中途中断
- 历史数据存在 only-job / no-paper 的记录

但 `server/schemas.py` 里的 `ProfileActivityItem` 对以下字段要求是必填 `str`：

- `paper_id`
- `paper_title`
- `paper_venue`
- `paper_pub_date`

因此接口返回前，FastAPI 在 response model 校验阶段抛出了 `ResponseValidationError`，最终表现为 500。

### 修改内容

修改文件：

- `server/database.py`
- `tests/test_profile_activity.py`

1. 在 `server/database.py -> list_profile_activity()` 中，对联表返回的空值做了默认值填充：

- 字符串字段 `None -> ""`
- 数值字段 `None -> 0 / 0.0`

重点覆盖：

- `paper_id`
- `paper_title`
- `paper_venue`
- `paper_pub_date`
- `paper_pdf_path`
- `paper_report_path`

2. 新增 `tests/test_profile_activity.py` 回归测试，覆盖“有 profile job 但没有 `papers` 记录”的场景。

## 4. Repo 校验与 Warning 渲染修复

### 问题现象

报告中出现如下 warning：

```text
论文正文提到了该仓库链接，但联网校验失败：code repository resolution 的所有候选模型均失败：gpt_pro -> BadRequestError: Error code: 400 - {'error': {'code': '400', 'type': 'invalid_params', 'message': 'Web Search cannot be used with JSON mode.'}}
```

同时，报告中的 GitHub Alert 语法未被正确渲染，页面上直接显示：

```md
> [!WARNING]
```

或中文翻译后的：

```md
> [!警告]
```

### 根因分析

1. 代码仓库联网校验阶段使用了 OpenAI Responses API 的 web search tool，但同时传入了 JSON mode：

- `tools=[{"type": "web_search_preview"}]`
- `response_format={"type": "json_object"}`

当前接口不支持这个组合，因此返回 `400 invalid_params`。

2. 报告前端只使用了 `remark-gfm`，并不原生支持 GitHub Alerts 扩展语法，因此 `[!WARNING]` / `[!警告]` 会原样显示，而不是渲染成醒目的 warning 块。

### 修改内容

修改文件：

- `modules/paper_interpreter/assembler.py`
- `utils/markdown.py`
- `web/src/pages/ReportViewer.tsx`

具体改动：

1. 在 `assembler.py` 的 code repository resolution 步骤中移除了 JSON mode，只保留 web search tool，避免 API 参数冲突。

2. 在 `utils/markdown.py` 中，新生成的报告不再输出 GitHub Alert 语法，而是直接输出普通 blockquote warning。

3. 在 `ReportViewer.tsx` 中增加 Markdown 预处理：

- 将历史报告中的 `> [!WARNING]`
- 以及翻译后出现的 `> [!警告]`

统一改写为普通 blockquote 标题，从而兼容旧报告和新报告。

### 验证

```bash

npm run build
```

结果：

- 后端目标文件编译通过

## 5. 报告页支持基于当前 PDF 重新生成报告

### 目标

在查看报告的界面中，直接对当前这篇论文重新发起一次报告生成，而不需要手动回到 Run 页重新上传 PDF。

### 修改内容

修改文件：

- `server/job_manager.py`
- `server/routers/jobs.py`
- `web/src/api/client.ts`
- `web/src/pages/ReportViewer.tsx`

具体改动：

1. 后端新增“基于已有 PDF 重新运行”的能力：

- 复用当前报告对应 job 的 `pdf_path`
- 复用原 job 的 `profile_id`
- 复用原 job 的 `config_snapshot`
- 以新的 manual job 重新进入完整 pipeline

2. 新增接口：

- `POST /api/jobs/{job_id}/rerun`

接口行为：

- 按 `job_id` 找到原 job
- 读取其关联 paper 记录与 PDF 路径
- 若 PDF 存在，则创建一个新的 rerun job

3. 在 `ReportViewer` 顶部工具栏增加 `Regenerate` 按钮：

- 仅在 job 维度报告页显示
- 点击后直接发起 rerun
- 成功后显示新的 job id，并提供进入 Run 页查看进度的入口

### 验证

已执行：

```bash

npm run build
```

结果：

- 后端目标文件编译通过

### 说明

- 当前按钮会新建一个 job，而不是覆盖旧报告

## 6. Reports 页支持失败任务重试并回填筛选条件

### 目标

在 `Reports` 历史页中，为失败的 auto job 提供重试入口，并在进入 `Run` 页后自动带入那次失败任务的筛选配置，允许先修改参数再重新跑。

### 修改内容

修改文件：

- `web/src/pages/ReportsPage.tsx`
- `web/src/components/JobHistoryCard.tsx`
- `web/src/pages/RunPage.tsx`

具体改动：

1. `Reports` 页中的失败 auto job 新增 `Retry` 按钮。

2. 点击 `Retry` 后会跳转到 `Run` 页，并携带以下失败任务配置：

- `topicName`
- `topicQuery`
- `topicKeywords`
- `track`
- `dateRangeDays`
- `classicMinCitations`
- `preferred venues`
- `profileId`

3. `Run` 页收到 retry draft 后会：

- 切换到 `auto` 模式
- 回填上述筛选条件
- 清空当前活动 job 绑定
- 清空手动上传文件状态
- 显示一条提示，说明当前配置来自哪一个失败 job

这样用户可以先调参，再手动点击 `Start Pipeline` 发起新的运行。
