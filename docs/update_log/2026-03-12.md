# 2026-03-12 更新日志

## 1. Method 解读要求增强

### 背景

现有报告的 `Method` 部分存在两个明显问题：

- 容易只概括“创新点”，但没有把关键术语讲清楚
- 没有交代输入数据进入模型后依次经过哪些模块，读者无法建立完整 pipeline 视角

以 TimeURL 这类论文为例，像 `random cropping`、`frequency-domain mixing` 这样的术语若只保留名字，不解释“它具体对数据做了什么”，解读价值会明显下降。

### 修改内容

修改文件：

- `modules/paper_interpreter/task_runner.py`

具体改动：

1. 强化 `_SECTION_REQUIREMENTS["method"]`

- 要求先给出 2-4 句端到端总览
- 明确说明输入是什么、按顺序经历哪些变换、最终输出什么
- 要求按执行顺序拆成 3-5 个核心模块/阶段
- 对重要机制、增强、表示、loss，不仅说明名称，还要解释：
  - 它字面上在做什么
  - 为什么引入
  - 与前后步骤如何衔接

2. 在 `_build_structured_task_instruction()` 中加入 method 专属硬性约束

- 至少保留一条 claim 专门概括完整 pipeline
- 关键术语必须用 plain language 定义，而不是只列名词
- 若出现 augmentation / fusion / decoder / loss 等，需要说明它位于哪一步、该步之后发生了什么变化
- 若论文提供架构图，要求解读与架构图对齐，帮助读者在脑中重建模型结构

3. 在 `_build_adjudication_messages()` 中收紧 method 裁决规则

- method 最多保留 5 条高价值 claims
- 裁决时优先保留“能重建 pipeline”和“能解释术语具体含义”的内容
- 避免最终答案退化成创新点口号式摘要

4. 将 method 的结构化输出进一步固定为三个主块

- `pipeline_overview`
- `modules`
- `training_objectives`

并在最终 Markdown 渲染阶段优先输出：

- `### Overall Pipeline`
- `### Key Modules`
- `### Training Objectives`

必要时再补一个 `### Key Takeaways`，保留证据锚点与高价值 claims。

5. 进一步把页面上的 Method 成文改为显式编号小节

为避免用户继续看到“还是一串散点 bullet、看不出分模块”，本轮又把渲染收紧为：

- `### 1. Overall Pipeline`
- `### 2. Key Modules`
- `#### Module 1: ...`
- `#### Module 2: ...`
- `#### Module 3: ...`
- `### 3. Training Objectives`

如果存在补充性的证据锚点摘要，则再追加：

- `### 4. Key Takeaways`

这样最终页面会直接看到按模块编号展开的结构，而不是只在 prompt 内部要求“模块化”。

### 结果

更新后，Method 部分的目标不再只是“总结 paper 有哪些设计”，而是进一步要求回答：

- 这个术语到底代表什么操作或对象
- 一条样本是如何一步步流过整个模型的
- 各模块在整体架构中分别承担什么职责
- 训练时到底在优化什么目标，以及这些目标如何影响最终行为

同时，最终成文也会更稳定地按模块分段，而不是只给一串松散 bullet。

### 验证

已执行：

```bash
python -m py_compile modules/paper_interpreter/task_runner.py
```

结果：

- 编译通过

## 2. Regenerate / Retry 改为替换式成功收敛

### 问题现象

现有 Web 端存在两个历史堆积问题：

1. 从报告页点击 `Regenerate` 后，会新建一个 job 和一份新报告，但旧报告及旧 job 仍然保留，导致同一篇论文出现多份历史报告。
2. 对失败 job 执行 `Retry` 后，即使新的任务最终成功，原来的失败记录依然保留在历史页里，观感上像“虽然修好了，但失败痕迹一直在”。

### 目标

改成“成功替换”语义：

- 同一篇文章的 regenerate 成功后，只保留新的那一份报告
- failed job 的 retry 成功后，旧 failed 记录自动消失
- 如果新的重跑本身失败，则旧记录不提前删除，仍然保留作为回退

### 修改内容

修改文件：

- `server/database.py`
- `server/job_manager.py`
- `server/routers/jobs.py`
- `server/schemas.py`
- `web/src/api/client.ts`
- `web/src/pages/RunPage.tsx`
- `web/src/pages/ReportViewer.tsx`
- `README.md`

具体改动：

1. 后端创建 job 链路新增 `replace_job_id`

- `POST /api/jobs`
- `POST /api/jobs/manual`
- `POST /api/jobs/{job_id}/rerun`

其中：

- `ReportViewer -> Regenerate` 会把当前报告对应的 `job_id` 作为替换来源
- `Reports -> Retry -> RunPage -> Start Pipeline` 也会把失败 job 的 `job_id` 一路传到创建接口

2. `JobManager` 在新 job **成功完成后** 执行替换清理

新增逻辑：

- 删除来源 job 的 Markdown 报告文件
- 删除来源 job 的 `results/jobs/{job_id}` 目录
- 删除来源 job 的 `data/fetch/jobs/{job_id}` 目录
- 删除来源 job 的数据库记录
- 删除仍然挂在该旧 job 下的 paper 记录

注意：

- 清理发生在新 job 成功之后，不在启动时做
- 因此如果重生成/重试失败，旧记录仍然在，不会误删

3. 前端创建 job 时透传替换来源

- `createJob()` 支持 `replace_job_id`
- `createManualJob()` 的 `FormData` 也支持 `replace_job_id`
- Run 页从 retry draft 启动新任务时会携带原 failed job id
- 新任务成功发起后会清空 retry banner 对应的 `retrySourceJobId`

4. Report Viewer 提示文案调整

- 当从报告页发起 regenerate 时，前端会明确提示：
  - 这是一个 replacement regeneration job
  - 成功后旧报告会被自动移除

### 结果

现在系统对这两类场景的观感变为：

- `Regenerate` 成功后，旧报告和旧 job 被新结果替换
- `Retry` 成功后，旧 failed job 自动消失
- 历史页最终更接近“当前有效状态”，而不是无限累积中间状态

### 验证

已执行：

```bash
python -m py_compile server/database.py server/job_manager.py server/routers/jobs.py server/schemas.py
```

结果：

- 编译通过
