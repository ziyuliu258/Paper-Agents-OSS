# 论文阅读上下文与记忆

> 本文已经不再是“未来计划”。它描述的是当前仓库已经落地的上下文与记忆实现，以及这些层如何影响 selector、interpreter、report refinement 和 Memory Workspace。

## 1. 当前上下文分层

Paper Agent 现在的上下文不是单一“大 prompt”，而是至少分成四层：

### 1.1 `paper_notes`

由 `build_paper_notes()` 统一生成，作为当前论文的共享结构化摘要。它是解释阶段最底层、最稳定的单篇论文上下文。

### 1.2 retrieval bundles

来自 Profile 级长期记忆。不同用途调用不同 bundle，而不是把 profile memory 粗暴拼成一大段文本。

### 1.3 WorkingMemory

这是一次 interpreter run 内部的 **job 级短期记忆**。它把 T1-T7、裁决、写回和报告展示串成同一条运行内认知链。

### 1.4 report artifacts 与 variants

解释完成后，短期记忆并不会消失，而是落成 artifact，并继续作为后续 report refinement 的 grounding 来源。

## 2. retrieval bundles 现状

当前已经正式使用四类 retrieval：

- `for_selector`
- `for_interpreter`
- `for_review_conflict`
- `for_translation_style`

它们分别服务于：

- auto 选论文
- T1-T7 解读
- 写回前冲突复核
- 中文翻译风格与术语一致性

其中 interpreter bundle 典型包含：

- `high_level_digest`
- `priority_claims`
- `relevant_evidence`
- `active_conflicts`
- `related_papers`

selector bundle 与 interpreter bundle 共享一部分高层认知，但呈现方式更偏“领域空白 / 争议 / 演化线索”。

## 3. WorkingMemory 现状

### 3.1 作用域

**单个 job / 单次 interpreter run**

### 3.2 当前主要字段

- `paper_notes`
- `retrieved_context`
- `task_outputs`
- `observations`
- `open_questions`
- `draft_claims`
- `terminology_map`
- `promotion_candidates`
- `metrics`

### 3.3 artifact stages

WorkingMemory 会在多个阶段落盘，当前可见阶段包括：

- `paper_notes_ready`
- `memory_context_ready`
- `tasks_complete`
- `writeback_ready`
- `report_assembled`

这些阶段会被 Report Viewer / RunPage 用来展示当前 job 到了哪一步。

### 3.4 当前职责

- 在 Group A / Group B / T7 之间共享中间状态
- 记录 retrieval footprint 与 prompt budget
- 为长期记忆写回前的 distillation 提供输入
- 为报告页提供“系统在本次运行中是如何形成理解”的可观测视图

## 4. T1-T7 与上下文组织

当前 `task_runner.py` 已把解释阶段收敛成共享上下文块 + 分组执行：

- Group A：`T1 -> T5 -> T6`
- Group B：`T2 -> T3 -> T4`
- T7：等待 A / B 完成后执行

每组任务共享统一的 `shared_context_block`，其中包含：

- 裁剪后的 `paper_notes`
- interpreter memory summary
- 结构化 section 的 schema example（仅 Group B）

另外：

- T2 → T3 → T4 会传 `prior_context`
- Method 输出会额外包含 `pipeline_overview / modules / training_objectives`

这意味着当前的“上下文优化”已经不只是 prompt 重构，而是任务结构、输出结构、WorkingMemory 和记忆写回一起联动。

## 5. Distillation 与 writeback

### 5.1 promotion funnel

长期记忆写回前，系统会先对 WorkingMemory 中的候选结论做 distillation，分类为：

- `accepted`
- `review_required`
- `rejected`

### 5.2 压缩策略

当前蒸馏摘要会在写盘前再次压缩，目标是：

- 保留事实和 evidence anchor
- 限制每条摘要与 overall assessment 的长度
- 减少“先生成再硬截断”的信息损失

### 5.3 review-conflict integration

写回前还会额外检索 `for_review_conflict` bundle，把：

- 邻近 claim
- active conflicts
- pending review

转成 `review_context` 注入 extraction prompt，帮助系统在写回时少制造与既有知识冲突的更新。

## 6. Memory V2 不变量

### 6.1 Profile 是隔离边界

不同 profile 之间的长期记忆互不混用。

### 6.2 英文是真值，中文是展示层

- entity / claim / evidence / synthesis 的英文字段是 source-of-truth
- 中文字段是自动翻译得到的 display layer
- 手工编辑英文后，中文会自动重译

### 6.3 manual lock

人工编辑过的对象会标记 `manual_locked`。AI 不直接覆盖，而是生成 review item。

### 6.4 provenance

每次写回都记录到 `memory_writebacks`，并绑定：

- profile
- job
- paper

因此当前支持：

- `delete_job_memories(profile_id, job_id)`
- `delete_paper_memories(profile_id, paper_id)`

删除时还会清理 orphaned claim / evidence / synthesis / entity / edge。

## 7. 双语展示与本地化

### 7.1 当前落盘产物

- `working_memory.json`
- `working_memory.zh.json`
- `distilled_memory_summary.md`
- `distilled_memory_summary.zh.md`

### 7.2 页面行为

以下页面采用整页统一语言切换：

- ReportViewer
- ProfileDetailPage
- MemoryWorkspacePage

若 localized artifact 缺失，则会回退英文原文。

## 8. report refinement 与记忆的关系

当前 report refinement 不是“纯 Markdown 改写”，而是受以下上下文共同约束：

- 当前报告正文
- `distilled_memory_summary.md`
- `working_memory.json`
- `report.en.md`
- 目标结构模式和 detail level

因此 report variants 可以理解为“基于现有解释过程的再编辑层”，而不是脱离原始运行上下文的自由生成。
