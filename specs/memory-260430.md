# Memory Workspace 与 Living Survey 交互修复规划（26-04-30）

## 1. 背景

本次规划针对 `specs/memory.md` 与 `ChangeLogs/26-04-29.md` 之后暴露的两类问题：

- Memory Workspace（记忆工作台）信息密度高，但入口和面板解释不足，导致新用户不知道先看什么、每块有什么用。
- Living Survey（动态综述）来源追溯暴露内部技术 key，且页面把大量派生内容平铺，读完后不容易形成领域理解。

本轮目标不是继续扩大 Memory V3 的数据结构，而是在现有 derived artifacts 基础上做**小而稳的可读性、来源展示和交互分层修复**。

## 2. 已完成代码调研结论

### 2.1 Memory Workspace 前端结构

相关文件：

- `web/src/pages/MemoryWorkspacePage.tsx`
- `web/src/api/client.ts`
- `utils/memory.py`
- `server/routers/memory_workspace.py`
- `server/schemas.py`

关键结论：

- `MemoryWorkspacePage.tsx` 当前有 11 个一级视图：`知识库`、`健康度`、`领域地图`、`证据矩阵`、`主题结构`、`知识空白`、`研究机会`、`图谱`、`时间线`、`冲突队列`、`修订历史`。
- 顶部 tab 平铺在一个 `flex flex-wrap` 容器中，没有按“阅读 / 证据治理 / 高级审计”分组。
- `知识库视图` 的 `整理摘要` 模式由 `snapshot.curated` 驱动，核心数据包括：
  - `domain_digest`
  - `priority_claims`
  - `active_conflicts`
  - `entity_clusters`
  - `source_bundles`
- `当前冲突与待确认默认解` 与 `高价值 Claims` 目前被放在同一个 `xl:grid-cols-2` 容器里，因此大屏并排显示。
- `实体分组` 来自 `utils/memory.py::_build_curated_sections()`，按 `entity_type` 聚合 Memory 图谱实体，展示每组最常被 claims 引用的实体。

### 2.2 Living Survey 前端结构与来源 bug

相关文件：

- `web/src/pages/LivingSurveyPage.tsx`
- `web/src/api/client.ts`
- `server/schemas.py`
- `server/routers/profiles.py`
- `utils/memory.py`

关键结论：

- `LivingSurveyPage.tsx` 直接把 `survey.sections` 全量平铺渲染。
- 来源追溯没有独立 formatter，直接在 JSX 中显示：`Theme: {block.theme_key}`、`Gap: {block.gap_key}`。
- 后端 theme key 的内部格式是 `theme:{anchor_key}`，例如 `theme:retrogfn`。
- 因此前端会显示 `Theme: theme:retrogfn`，既重复 `Theme`，又暴露内部 slug。
- gap key 也有类似风险，例如 `coverage:theme:retrogfn` 会被直接展示。
- 当前 `LivingSurveyBlock` schema 只有扁平字段：`theme_key`、`gap_key`、`claim_ids`、`synthesis_ids`、`review_ids`、`paper_ids`，没有结构化 provenance display label。

### 2.3 后端 Memory / synthesis / survey 逻辑

相关文件：

- `utils/memory.py`
- `modules/paper_interpreter/distillation.py`
- `tests/test_memory_workflow.py`

关键结论：

- `领域认知摘要` 不是从 claims 直接生成，而是读取 `memory_synthesis_items` 后按 `consensus / debate / evolution / open_question` 分组。
- 当前只显示存在数据的 synthesis 类型，不为空类型显示占位解释。
- AI4Chem 只出现 `开放问题`，原因大概率是该 profile 当前有效 synthesis 里只有 `open_question`，没有有效的 `consensus / debate / evolution`。
- `rebuild_profile_cognition()` 会保守地派生 `debate / open_question / evolution`，但不会从所有 supported findings 自动派生 `consensus`，这是为了避免把单篇或薄证据结论过度提升为共识。
- Living Survey 的 `theme_key` 是稳定关联键，不应直接改 key 语义；应新增展示层格式化或补充 display label。

## 3. 用户问题逐条判断

### 3.1 工作台冲突面板不应全量展示

判断：应修复。

当前 `当前冲突与待确认默认解` 在摘要视图里展示每条 pending review，和右侧 `高价值 Claims` 并排，导致摘要页视觉负担大。

规划改法：

- 将 `当前冲突与待确认默认解` 改为全宽单行治理提示卡。
- 卡片只显示待解决冲突数量、是否含建议更新、对后续 Agent 默认解的影响说明。
- 保留 `前往冲突队列` 按钮，具体冲突仍在 `冲突队列` 视图中处理。
- 不在摘要视图中全量展开每条 review。

### 3.2 高价值 Claims 应单独全宽放在下面

判断：应修复。

规划改法：

- 将 `高价值 Claims` 从并排 grid 中拆出来，单独作为全宽面板。
- 保留当前“默认折叠、点击展开一条证据预览”的机制。
- 不改 `priority_claims` 后端排序规则。

### 3.3 实体分组有什么用

解释：

- `实体` 是 Memory 图谱里的稳定概念节点，不是论文作者或普通命名实体。
- `实体分组` 按类型聚合这些节点，例如任务、方法、模块、数据集、指标、概念。
- 它的价值是帮助用户快速判断当前 profile 的知识覆盖结构：这个方向主要围绕哪些任务、方法和数据集展开，哪些类型明显缺失。
- 每个实体后面的数字代表它关联的 claim 数，可理解为该实体在当前记忆中的“被引用强度”。
- 如果需要逐条改实体名称、类型或摘要，应进入 `原始对象与编辑`。

规划改法：

- 保留 `实体分组`，但补充更明确的中文说明。
- 将文案从“避免实体列表越积越长”改为“这是 Memory 图谱节点的类型聚合，用于判断任务/方法/数据集/指标覆盖是否均衡”。
- 在卡片内补一句说明：实体标签后的数字是关联 claims 数。

### 3.4 为什么领域认知摘要只有开放问题

解释：

- `领域认知摘要` 只展示已经写入 `memory_synthesis_items` 的高层认知。
- 如果当前 profile 没有 `consensus / debate / evolution` 类型的 synthesis，就只会显示 `open_question`。
- 这是保守设计，不自动把普通 supported finding 提升为共识，避免长期记忆过度自信。

规划改法：

- 不自动生成伪共识，不改变后端 synthesis 派生规则。
- 在 `领域认知摘要` 顶部增加解释：这里展示的是高层 synthesis，不是全部 claims。
- 当只存在部分类型时，显示一条轻量说明，提示“当前尚未形成共识/争议/演化类高层认知；可从高价值 Claims 或后续论文写回中逐步沉淀”。
- 可在 UI 中用缺失类型提示解释为什么这里只有开放问题，而不是让用户误以为页面漏数据。

### 3.5 工作台 tab 过多且缺少合理中文注释

判断：应修复，但避免大规模重构。

规划改法：

- 不隐藏现有入口，避免用户找不到功能。
- 将一级视图入口按用途分组展示：
  - `先读懂领域`：知识库、领域地图、主题结构、研究机会。
  - `查证据与问题`：证据矩阵、知识空白、健康度、冲突队列。
  - `高级审计`：图谱、时间线、修订历史。
- 每个分组增加一句中文说明，告诉用户这个分组解决什么问题。
- 保持 `activeView`、路由和数据加载逻辑不变，只改入口组织与文案。
- 在默认 `知识库视图` 的说明中强化推荐阅读路径：先看领域认知摘要，再看冲突计数，再看高价值 Claims，最后按需进入实体/来源或原始编辑。

### 3.6 Living Survey 来源追溯显示 `Theme: theme:retrogfn`

判断：这是明确 bug，应修复。

规划改法：

- 不改变后端 `theme_key` / `gap_key` 的内部 key 语义。
- 前端新增轻量格式化函数，至少把：
  - `theme:retrogfn` 显示为 `RetroGFN` 或 `retrogfn` 的可读形式。
  - `coverage:theme:retrogfn` 显示为 `Coverage · RetroGFN` 或同等可读形式。
- 如果后端已经能提供 block title，则优先用 block title 展示当前主题/空白；key 只作为兜底。
- 来源追溯标签改为中文：`主题`、`知识空白`、`Claims`、`高层认知`、`审阅项`、`论文来源`。

### 3.7 Living Survey 信息多但读完没有抓手

判断：应修复。

规划改法：

- 页面顶部增加“如何阅读这页”的引导卡，说明推荐顺序：
  - 先看领域概览。
  - 再看研究主题。
  - 接着看开放问题与研究机会。
  - 最后展开来源追溯，去 Workspace 审查原始对象。
- 顶部统计卡片中文化并解释用途：论文来源、研究主题、追踪空白、生成时间。
- `目录与来源说明` 不再只显示 `N blocks`，改成 `N 条内容`，并为每个章节显示一句用途说明。
- 每个 section 增加“这一节有什么用”的说明，避免只是堆卡片。
- 每个 block 的来源追溯增加操作入口：
  - 有 claims / synthesis / reviews / papers 时，提供 `打开工作台查看来源对象`。
  - 当前先使用统一 Workspace 跳转，不做复杂 deep-link，避免改动过大。
- 统一中文模式下的页面 chrome 文案，减少 `Paper Sources / Themes / Tracked Gaps / Generated / blocks` 这类混杂英文。

## 5. 详细修复规划

### 5.1 Memory Workspace 入口分组

执行步骤：

1. 将 `tabs` 从单数组改为分组数组，仍保留原 `WorkspaceView` id。
2. 渲染为 3 个分组区域，每组显示标题、说明和按钮。
3. 保持按钮点击仍调用 `setActiveView(tab.id)`。
4. 不改变任何 lazy loading 条件。

验收标准：

- 所有原有视图入口仍可点击。
- 用户能从分组文案理解每组用途。
- 移动端按钮能换行，不横向溢出。

### 5.2 Knowledge curated 面板重排

执行步骤：

1. 将 `当前冲突与待确认默认解` 与 `高价值 Claims` 的外层 `xl:grid-cols-2` 改为单列上下结构。
2. 将冲突面板改为摘要卡：显示 pending review 数量、建议更新数量和治理说明。
3. 保留 `前往冲突队列` 操作。
4. `高价值 Claims` 保持全宽，保留点击展开证据预览。

验收标准：

- 摘要页不再全量列出 pending review。
- `前往冲突队列` 可进入具体处理视图。
- `高价值 Claims` 单独全宽显示，展开态正常。

### 5.3 实体分组说明增强

执行步骤：

1. 更新 `实体分组` 的 `SectionTitle` hint。
2. 更新 `CardDescription`，说明实体是 Memory 图谱节点。
3. 在实体列表上方增加一行解释：标签后的数字为关联 claims 数。

验收标准：

- 用户能看懂实体分组的用途。
- 不改变 `entity_clusters` 数据来源和跳转逻辑。

### 5.4 领域认知摘要缺失类型说明

执行步骤：

1. 在 `领域认知摘要` 增加说明：它展示高层 synthesis，不是所有 claims。
2. 根据当前 `curated.domain_digest` 的 `section_type` 计算缺失类型。
3. 如果缺少 `consensus / debate / evolution / open_question` 中的部分类型，显示轻量提示。
4. 文案强调“缺失代表当前尚未形成可提升的高层认知”，不暗示系统错误。

验收标准：

- AI4Chem 只有开放问题时，页面能解释为什么没有其他认知摘要。
- 不自动伪造 synthesis。

### 5.5 Living Survey 来源追溯格式化

执行步骤：

1. 在 `LivingSurveyPage.tsx` 新增纯函数格式化来源 key。
2. 处理 `theme:`、`coverage:theme:`、snake/kebab/slug 和常见大小写。
3. 来源追溯展示中文标签：`主题`、`知识空白`、`Claims`、`高层认知`、`审阅项`、`论文来源`。
4. 当没有结构化来源时，保留清晰空状态。

验收标准：

- 不再显示 `Theme: theme:retrogfn`。
- `theme:retrogfn` 至少显示为 `RetroGFN`。
- `coverage:theme:retrogfn` 至少显示为 `Coverage · RetroGFN` 或中文等价展示。

### 5.6 Living Survey 阅读引导和目录优化

执行步骤：

1. 顶部标题仍保留 Living Survey，但增加中文副标题说明它是“面向阅读的动态领域综述”。
2. 统计卡片中文化，并补充 hint。
3. 新增“建议阅读路径”卡片，说明先后顺序。
4. 目录中将 `blocks` 改为 `条内容`，并按 section key 显示用途说明。
5. 每个 block 的来源追溯下增加统一的 Workspace 入口。

验收标准：

- 新用户能知道这页用于快速读领域结构、定位空白、追溯来源。
- 页面仍保持高信息密度，但不会只是平铺列表。
- 中文模式下页面 chrome 不再大量混用英文。

## 7. 风险与处理

| 风险 | 处理 |
|---|---|
| 只做前端格式化无法拿到真正主题标题 | 先使用 block title 与 key formatter 兜底；如仍不足，再补后端可选 display 字段。 |
| tab 分组后用户误以为功能减少 | 不隐藏任何入口，只分组展示。 |
| 冲突摘要隐藏了细节 | 摘要只放在知识库首页，具体处理仍在冲突队列。 |
| 领域认知摘要缺失类型被理解为数据丢失 | 用文案明确说明这是高层 synthesis 的保守提升结果。 |
| 后端缓存导致 Living Survey 旧字段不刷新 | 若只改前端展示，无需清缓存；若改后端生成逻辑，需要升级 artifact version 或处理旧缓存兜底。 |

## 8. 推荐实施策略

推荐先执行**前端优先的最小修复**：

1. Memory Workspace 做入口分组、冲突摘要、高价值 Claims 全宽、实体解释、认知摘要缺失说明。
2. Living Survey 做来源 key 格式化、中文化、阅读路径、目录说明、Workspace 入口。
3. 跑 `npm run build` 与 `npm run lint`。
4. 使用 subagents 做独立前端 review。

如果 review 或真实页面验收发现来源追溯仍然不可读，再进入第二阶段：后端新增可选 provenance display 字段，并补后端测试。
