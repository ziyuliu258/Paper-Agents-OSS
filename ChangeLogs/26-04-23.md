# 2026-04-23 Runtime Source 收口修复

## 背景与目标

在 2026-04-22 把 runtime source 与 `.env` 密码保护解耦后，新的设计虽然避免了“未解锁时返回空配置”这类高耦合问题，但也留下了一个新的语义缺口：

- `ENV_RUNTIME_ACCESS_GUARD=off` 时，前端或自定义客户端仍可通过 browser runtime override 抢占 effective runtime；
- runtime access 校验看的 source，与 `load_runtime_config()` 真正使用的 source 不是同一个判定点；
- 仓库文档仍默认把 browser runtime 描述为始终可用，和新的安全预期不一致。

本次修复目标是用最小改动把 effective runtime source 收口成统一真相点，并明确新的自动选源规则：

- `ENV_RUNTIME_ACCESS_GUARD=off`：后端统一只使用服务器 `.env`
- `ENV_RUNTIME_ACCESS_GUARD=password` 且当前浏览器已解锁：优先使用服务器 `.env`
- `ENV_RUNTIME_ACCESS_GUARD=password` 且当前浏览器未解锁：回落到当前浏览器的 runtime settings；浏览器也没有可用 override 时才拒绝请求

## 影响范围

- 后端 runtime source 判定：`utils/config.py`、`utils/runtime_access.py`
- runtime access API / schema：`server/routers/config.py`、`server/schemas.py`
- 前端 Settings 状态展示：`web/src/api/client.ts`、`web/src/pages/SettingsPage.tsx`
- 运行时回归测试：`tests/test_runtime_config.py`
- 文档：`README.md`、`web/README.md`、`AGENTS.md`、`specs/Web工作台与记忆系统-现行规格.md`

## 核心改动

### 1. effective runtime source 收口

- 新增 `is_browser_runtime_enabled()`，把 browser runtime 是否启用明确绑定到 `ENV_RUNTIME_ACCESS_GUARD=password`
- 新增 `get_effective_runtime_source()`，统一判定本次请求最终应使用 `.env`、browser override 还是无可用来源
- `has_runtime_config_for_request()`、`load_runtime_config()`、`get_missing_runtime_config_detail()` 全部改为依赖同一个 effective source

结果：

- `guard=off` 时，browser override 即使存在也会被忽略，effective runtime 统一退回 `.env`
- `guard=password` 且当前浏览器已解锁时，优先使用 `.env`
- `guard=password` 且当前浏览器未解锁时，若存在 override 就回落到 browser runtime

### 2. runtime access 响应增加 browser 能力位

- `GET /api/config/runtime/access` 新增 `browser_runtime_enabled`
- 前端不再需要自己把 `guard=password` 硬编码解释为“browser runtime 可用”

### 3. Settings 页最小对齐

- Settings 页不再让用户手动选择 `env/browser` 模式，而是明确展示自动选源规则
- browser settings 仍允许保存在本地，但 UI 会明确提示它们只在 `.env` 不可用时作为 fallback 生效
- `/api/config/runtime/access` 会带上当前浏览器 session 的 env auth token，以便正确显示已解锁状态

### 4. 回归测试补齐

- 新增 `guard=off` 时 browser override 被忽略的测试
- 新增 `guard=off` 且 `.env` 缺失时，browser override 不能救场的测试
- 新增 `guard=password` 且未解锁时 browser override 作为 fallback 生效的测试
- 新增 `guard=password` 且已解锁时 `.env` 优先覆盖 browser override 的测试
- 新增 `/api/config/runtime/access` 的 `browser_runtime_enabled` 断言

## 数据结构 / API / 页面变化

- `RuntimeAccessResponse` 新增：
  - `browser_runtime_enabled`
- Settings 页 runtime source 区域现在会根据该字段展示自动选源规则与当前解锁状态

## 验证结果

前端：

```bash
conda activate P-Agent && cd web && npm run lint
conda activate P-Agent && cd web && npm run build
```

结果：

- `build` 通过
- `lint` 无 error，仍保留 1 个既有 warning：`web/src/pages/RunPage.tsx` 的 `react-hooks/exhaustive-deps`

后端：

```bash
conda activate P-Agent && python -m py_compile utils/config.py utils/runtime_access.py server/schemas.py server/routers/config.py tests/test_runtime_config.py
```

结果：通过。

补充说明：

- 当前 `P-Agent` 环境缺少 `pytest`，无法直接执行 `python -m pytest tests/test_runtime_config.py`
- 同时该环境也缺少 `fastapi`，因此 `python -m unittest tests.test_runtime_config` 无法在当前环境完成导入执行
- 已保留并更新 `tests/test_runtime_config.py`，待依赖齐全后可直接执行该定向回归

## 兼容性与约束

- 这次改动有意改变一条关键语义：`ENV_RUNTIME_ACCESS_GUARD=off` 不再只是“关闭 `.env` 密码层”，也同时表示“后端不启用 browser runtime override”
- browser runtime settings 仍会保留在浏览器本地，不会被自动清空；只是当 server guard 为 `off` 时不会参与后端求值
- 当 `ENV_RUNTIME_ACCESS_GUARD=password` 且当前浏览器未解锁时，browser runtime settings 现在会作为自动 fallback 来源，而不是再要求用户手动切模式
- 本次没有移除 `GET/PUT /api/config/runtime`，因为它仍是历史接口面，但当前 active runtime 继续不依赖该文件

## 清理情况

- 未新增临时脚本
- 未修改与本次任务无关的下游 LLM / embedding / selector 实现
- 未清理用户现有的浏览器本地 runtime settings，仅调整其启用条件
