# 2026-04-22 Memory V3 更新记录

## 背景与目标

本次更新的目标是把记忆模块从原有的 Memory V2 能力扩展到可落地的 Memory V3：

- 补齐 claim relation / opportunity 等派生层；
- 在不破坏 V2 主功能的前提下完成懒迁移与兼容；
- 接通前端只读机会视图；
- 用真实 LLM API 对新 memory 抽取链路做一次端到端验证；
- 清理验证阶段产生的临时脚本与缓存文件。

## 影响范围

- 后端记忆数据层：`utils/memory.py`
- Memory V3 新增纯函数模块：`utils/memory_claim_relations.py`、`utils/memory_opportunities.py`
- 后端 API / schema：`server/schemas.py`、`server/routers/memory_workspace.py`、`server/routers/profiles.py`
- 前端类型与页面：`web/src/api/client.ts`、`web/src/pages/ProfileDetailPage.tsx`、`web/src/pages/MemoryWorkspacePage.tsx`
- 配置兼容：`utils/config.py`、`utils/llm.py`
- 测试：`tests/test_memory_v3_schema_migration.py`、`tests/test_memory_v3_claim_relations.py`、`tests/test_memory_v3_opportunity_routes.py`

## 核心改动

### 1. Memory V3 数据结构升级

- 记忆 schema 升级到 `v4`
- `memory_claims` 新增：
  - `scope_json`
  - `stability_score`
  - `last_supported_at`
  - `last_challenged_at`
- `memory_claim_evidence` 新增：
  - `anchor_kind`
  - `context_before`
  - `context_after`
  - `structured_signal_json`
- `memory_profile_state` 新增：
  - `claim_relations_stale`
  - `claim_relations_updated_at`

### 2. Memory V3 派生层

- 新增 `claim_relations` 派生层：
  - 使用确定性规则生成 `reinforces / extends / contradicts`
  - 结合 overlap、entity、stance、review 状态估计 `stability_score`
- 新增 `opportunity_snapshot` 派生层：
  - 从 theme / gap / claim_relations / review 中抽取只读研究机会
  - 当前机会项包括：
    - `persistent_contradiction`
    - `thin_evidence_high_impact`
    - `consensus_boundary_missing`
    - `cross_theme_transfer`

### 3. 真实 LLM 抽取链路增强

- Memory extraction prompt 明确要求输出：
  - claim `scope`
  - evidence `anchor_kind`
  - evidence `context_before` / `context_after`
  - evidence `structured_signal`
- parser 和 writeback 已完整接线到数据库
- 真实 API 烟测确认：
  - `scope_json` 不再全空
  - evidence 的 `anchor_kind` 和 `structured_signal_json` 已真实落盘

### 4. Lite 模型配置兼容修复

- `.env` 兼容同时支持：
  - `LITE_MODEL`
  - `LITE_MODLE`
- 修复 `call_lite_model()` 中 alias 读取方式，避免未配置时错误回退到字面量 `lite_model`

### 5. 前端视图补齐

- `ProfileDetailPage` 显示 `opportunity_preview`
- `MemoryWorkspacePage` 新增只读 `研究机会` 视图：
  - 展示机会类型、优先级、reason codes、risk flags、建议验证步骤
  - 支持跳转到原始 claim / review queue

## 补充修复：Memory V3 接线与一致性修正

### 背景与目标

在完成 V3 主功能后，又对接入可靠性做了一轮跨层修复，目标是解决以下真实缺口：

- memory 写回失败会被静默吞掉，可能出现“报告成功但 memory 缺失”；
- profile 删除链路未显式清理 V3 新增的 `memory_claim_relations` / `memory_derived_artifacts`；
- 手工编辑与 review adopt 仍停留在 V2 子集，`scope` 与 structured evidence 字段无法完整保真；
- manual lock 对 claim / synthesis 的保护不完整，AI 候选仍可能渗透修改 evidence、entity links 或 synthesis claim links；
- theme / gap / opportunity 构建依赖截断 basis，大 profile 下存在派生视图不完整风险；
- gap / opportunity 的 theme badge 在中文页面下本地化不统一。

### 影响范围

- `modules/paper_interpreter/agent.py`
- `utils/memory.py`
- `utils/memory_opportunities.py`
- `server/schemas.py`
- `web/src/api/client.ts`
- `web/src/pages/MemoryWorkspacePage.tsx`
- `web/src/pages/ProfileDetailPage.tsx`

### 核心改动

- memory 写回从 best-effort 改为主流程失败即失败，不再静默跳过；
- profile 删除时显式清理 `memory_claim_relations` 与 `memory_derived_artifacts`；
- `save_claim()` / `save_evidence()` / review adopt 补齐对 `scope`、`anchor_kind`、`context_before`、`context_after`、`structured_signal` 的支持；
- manual-locked claim 在 AI candidate update 分支下不再继续追加 evidence 或覆盖 entity links；
- manual-locked synthesis 在 AI candidate update 分支下不再改写 claim linkage；
- claim relation rebuild 和派生 basis 改为全量读取，不再依赖固定小窗口 limit；
- `list_claims()` 去掉 N+1 entity 查询，改为批量聚合 claim-entity 映射；
- opportunity prompt/schema 明确补入 `domain_facts` 输出要求；
- gap / opportunity 暴露 `theme_title_localized` / `theme_titles_localized`，前端按当前语言展示。

## 补充修复：Runtime Settings 与密码层解耦

### 背景与目标

本次更新聚焦于修复 runtime settings / `.env` 密码保护的高耦合问题。旧实现把“运行时来源选择”和“服务器 `.env` 访问保护”混在同一层，导致：

- 本地自用与本地调试会被密码层误伤；
- `load_runtime_config()` 在未解锁时返回空配置，错误语义不清；
- Settings 页必须从 runtime config 视图里顺带读取密码状态，前后端边界不清晰；
- 部署时无法通过一个明确开关启用统一保护。

目标是把密码层抽成独立 access guard：本地默认关闭，对外部署时只需切一个参数即可启用。

### 影响范围

- 新增独立后端模块：`utils/env_runtime_access.py`、`utils/runtime_access.py`
- runtime source 解析回收：`utils/config.py`
- 中间件与 API：`server/app.py`、`server/routers/config.py`、`server/routers/jobs.py`、`server/routers/reports.py`、`server/schemas.py`
- 前端 transport / settings：`web/src/lib/runtimeSettings.ts`、`web/src/api/client.ts`、`web/src/pages/SettingsPage.tsx`
- 脚本与测试：`scripts/generate_env_runtime_password_hash.py`、`tests/test_runtime_config.py`
- 文档：`README.md`、`web/README.md`、`AGENTS.md`

### 核心改动

- 新增独立 server env access guard：
  - `ENV_RUNTIME_ACCESS_GUARD=off|password`
  - 默认 `off`
  - 仅当值为 `password` 时，`ENV_RUNTIME_PASSWORD_HASH` 才真正生效
- 新增 `utils/env_runtime_access.py` 承载：
  - guard mode 解析
  - password hash 元数据
  - challenge / verify / token 解析
  - access snapshot 与拒绝原因
- `utils/config.py` 退回 runtime source 职责：
  - 只负责 browser override / server env 的配置解析
  - `load_runtime_config()` 不再因为未解锁而回退空配置
- 新增 `utils/runtime_access.py` 统一 router 准入判断：
  - 先判断是否有 runtime source
  - 再判断该 source 是否被 guard 放行
- 新增独立接口 `GET /api/config/runtime/access`：
  - Settings 页不再从 `/api/config/runtime` 读取密码层状态
- Settings 前端状态流解耦：
  - provider 配置视图继续走 runtime config
  - 密码层状态单独走 runtime access status
  - `loadLocalRuntimeMode()` 去掉“残留 browser settings 自动切 browser”的隐式推断，默认回到 `env`

### 数据结构 / API / 页面变化

- `RuntimeEnvModeStatus` 新增：
  - `guard_mode`
  - `password_configured`
- 新增响应模型 `RuntimeAccessResponse`
- `EnvRuntimeUnlockChallengeResponse` / `EnvRuntimeUnlockVerifyResponse` 增加：
  - `guard_mode`
  - `password_configured`
- 新接口：
  - `GET /api/config/runtime/access`
- Settings 页文案改为明确区分：
  - `Runtime Mode`
  - `Server Env Access`
  - `Open Server Env Access`
  - `Protected Server Env Access`
